## 1. 技术栈选型

### 1.1 总体技术架构原则

基于"约定优于配置"的设计理念，选择成熟、稳定、生态完善的技术栈，确保：
- 高并发处理能力
- 数据一致性保证
- 优雅的API设计
- 完善的回滚机制

### 1.2 后端技术栈

| 层级 | 技术选型 | 选型理由 |
|-----|---------|---------|
| **编程语言** | Go 1.22+ | 高性能、并发模型优秀、部署简单、内存占用低 |
| **Web框架** | Gin | 高性能、路由简洁、中间件生态完善、约定优于配置 |
| **ORM层** | GORM | 约定优于配置、自动迁移、关联查询强大 |
| **API文档** | Swagger/OpenAPI 3.0 | 自动生成文档、在线调试、前后端协作友好 |
| **配置管理** | Viper | 多格式支持、环境变量覆盖、热重载 |
| **日志** | Zap + Lumberjack | 高性能、结构化日志、日志轮转 |
| **验证器** | go-playground/validator | 标签化验证、自定义验证器、国际化支持 |

### 1.3 数据库技术栈

| 类型 | 技术选型 | 选型理由 |
|-----|---------|---------|
| **关系型数据库** | PostgreSQL 15+ | 强大的事务支持、JSON字段、窗口函数、并发控制优秀 |
| **缓存** | Redis 7+ | 高性能、丰富数据结构、持久化支持、集群方案成熟 |
| **全文搜索** | PostgreSQL tsvector | 轻量级、与业务数据同库、避免数据同步复杂度 |

### 1.4 消息队列与异步处理

| 功能 | 技术选型 | 选型理由 |
|-----|---------|---------|
| **消息队列** | Redis Stream | 轻量级、持久化、消费组支持、与缓存同技术栈 |
| **定时任务** | Asynq | 基于Redis、分布式、任务重试、优先级支持 |
| **工作流引擎** | Temporal (备选) | 状态持久化、事件溯源、长时运行流程（复杂审批场景） |

### 1.5 安全与认证

| 功能 | 技术选型 | 选型理由 |
|-----|---------|---------|
| **认证** | JWT + Refresh Token | 无状态、跨域友好、刷新机制完善 |
| **权限控制** | Casbin | RBAC/ABAC支持、策略持久化、性能优秀 |
| **数据加密** | AES-GCM | 标准算法、认证加密、Go标准库支持 |
| **密码哈希** | Argon2id | 抗彩虹表、内存硬化、现代推荐算法 |

### 1.6 基础设施与部署

| 功能 | 技术选型 | 选型理由 |
|-----|---------|---------|
| **容器化** | Docker + Docker Compose | 环境一致性、部署简单、本地开发友好 |
| **反向代理** | Nginx | 高性能、负载均衡、静态文件服务 |
| **监控** | Prometheus + Grafana | 时序数据库、丰富仪表盘、告警机制 |
| **链路追踪** | OpenTelemetry + Jaeger | 标准协议、分布式追踪、性能分析 |

## 3. 数据库Schema设计

### 3.1 数据库设计原则

1. **一致性优先**：所有写操作使用数据库事务，明确回滚方案
2. **软删除**：重要业务数据使用软删除，保留操作痕迹
3. **审计字段**：所有表包含创建时间、更新时间、创建人、更新人
4. **乐观锁**：使用版本号实现乐观锁，防止并发更新冲突
5. **JSON字段**：灵活属性使用JSONB存储，支持索引查询

## 5. 安全架构设计

### 5.1 认证机制

采用JWT + Refresh Token双Token机制：

1. **访问令牌（Access Token）**：短期有效（2小时），用于API访问
2. **刷新令牌（Refresh Token）**：长期有效（7天），用于刷新访问令牌
3. **令牌存储**：
   - Access Token：内存存储，不持久化
   - Refresh Token：HttpOnly Cookie存储，防止XSS攻击
4. **令牌黑名单**：Redis存储已失效的Token，支持主动登出

### 5.2 权限控制

基于Casbin实现RBAC（基于角色的访问控制）+ ABAC（基于属性的访问控制）：

```go
// RBAC策略示例
p, role:project_manager, project, create, allow
p, role:project_manager, project, read, allow
p, role:project_manager, project, update, allow
p, role:department_manager, project, read, allow, department == user.department

// ABAC策略示例
p, role:member, task, update, allow, task.assignee == user.id
```

### 5.3 数据安全

1. **传输加密**：全站HTTPS，TLS 1.3
2. **存储加密**：
   - 密码：Argon2id哈希存储
   - 敏感字段：AES-GCM加密存储
3. **数据脱敏**：日志、接口返回时对敏感信息脱敏
4. **审计日志**：所有操作记录审计日志，包含操作人、时间、IP、操作内容、变更前后数据

### 5.4 API安全

1. **限流**：基于Redis的令牌桶算法，防止接口滥用
2. **XSS防护**：输入验证、输出编码
3. **CSRF防护**：SameSite Cookie + CSRF Token
4. **SQL注入防护**：使用GORM参数化查询
5. **IDOR防护**：权限验证时同时验证资源所有权

## 6. 事务与一致性设计

### 6.1 事务边界

```go
// 事务使用示例
func (s *ProjectService) CreateProject(ctx context.Context, req *CreateProjectRequest) (*Project, error) {
    tx := s.db.WithContext(ctx).Begin()
    defer func() {
        if r := recover(); r != nil {
            tx.Rollback()
        }
    }()

    // 1. 创建项目
    project := &Project{...}
    if err := tx.Create(project).Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    // 2. 创建立项记录
    initiation := &ProjectInitiation{ProjectID: project.ID, ...}
    if err := tx.Create(initiation).Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    // 3. 初始化项目预算
    budget := &ProjectBudget{ProjectID: project.ID, ...}
    if err := tx.Create(budget).Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    // 4. 创建审计日志
    auditLog := &AuditLog{...}
    if err := tx.Create(auditLog).Error; err != nil {
        tx.Rollback()
        return nil, err
    }

    if err := tx.Commit().Error; err != nil {
        return nil, err
    }

    return project, nil
}
```

### 6.2 分布式事务

对于跨服务操作，采用最终一致性方案：
1. **事件驱动**：基于Redis Stream的事件总线
2. **补偿机制**：明确的回滚操作
3. **重试策略**：指数退避重试，最大重试次数
4. **死信队列**：处理失败的消息

## 7. 性能优化策略

### 7.1 缓存策略

1. **多级缓存**：
   - L1：本地内存缓存（bigcache），热点数据
   - L2：Redis缓存，共享数据
2. **缓存失效**：
   - 写操作更新缓存（Cache-Aside模式）
   - 合理的TTL设置
   - 缓存预热
3. **缓存穿透**：布隆过滤器
4. **缓存击穿**：互斥锁
5. **缓存雪崩**：随机TTL

### 7.2 数据库优化

1. **索引优化**：合理创建索引，定期分析索引使用情况
2. **查询优化**：避免N+1查询，使用预加载
3. **读写分离**：主从复制，读操作走从库
4. **分区表**：大表按时间或业务分区
5. **连接池**：合理配置数据库连接池

### 7.3 API优化

1. **数据压缩**：Gzip压缩响应
2. **分页查询**：避免全量返回
3. **字段筛选**：支持请求方指定返回字段
4. **批量操作**：提供批量接口，减少请求次数
5. **CDN加速**：静态资源CDN分发
